Concept Manual / Source-backed

把 Agent Memory Hub 的每个环节、每个名词、每条公式讲清楚。

这份页面不是宣传页，而是“读源码后的概念说明书”。它按真实数据流解释 AMH： 证据怎么进来，MemoryItem 怎么形成，索引怎么投影，召回怎么排序，防火墙怎么阻断，分层上下文怎么装载，反馈和 Loop 怎么把下一轮变得更可靠。

先抓住一个总模型

原始证据 Evidence
  -> 候选 Candidate / Pending / Review
  -> WriteService 写入漏斗
  -> MemoryItem 长期事实源
  -> Index / Graph / Runtime 派生投影
  -> Retrieval 候选召回
  -> Context Firewall 注入前门禁
  -> ContextPack 分层装载
  -> Feedback / Governance / Loop 回写治理
证据不等于记忆 索引不等于事实源 召回不等于注入 完成不等于有证据

这份详解解决什么困惑

  • 为什么召回前要先讲维护。
  • 召回出来的到底是 item、正文、摘要，还是路径。
  • FTS、BM25、向量、RRF、MMR、rerank、Hopfield 各自负责什么。
  • confidence、decay、maturity、feedback 的来源和公式。
  • Loop 在系统里不是排序算法，而是复杂任务账本。
  • 哪些能力是事实源，哪些只是派生读模型或候选。
00 / One Sentence

AMH 的核心不是“存”和“搜”，而是“可信上下文闭环”。

如果只说“记忆召回”，读者会自然理解成一个 RAG 小工具。AMH 的复杂度在于它把来源、写入、索引、召回、注入、反馈、治理和任务账本连在一起。

1Evidenceprompt、transcript、文件、资源、运行事件。
2Candidate候选、pending、review，暂不等于事实。
3WriteService审计、字段增强、质量检查、证据 sidecar。
4MemoryItem长期可复用结论，落在 items/。
5IndexFTS、向量、metadata、refs_graph。
6Maintenancematurity、drift、summary rewrite、repair。
7RetrievalSearchFilter、BM25、vector、RRF、rerank。
8Firewallinclude、demote、exclude。
9ContextPacklocator、overview、detail、detail_uri。
10Feedbackadopted/rejected、Loop、治理回写。
01 / Object Model

先分清系统里的对象，否则后面所有链路都会混。

AMH 里最重要的不是某个算法，而是对象边界。原始证据、长期记忆、索引投影、运行账本、候选队列、上下文包分别承担不同责任。

对象 一句话定义 作用 不是 常见位置
Evidence 原始发生过的事实材料。 回答“这件事有没有来源，来源在哪”。 不是长期可复用结论。 sources/、resources/、extractions/
Conversation 原始对话消息 JSONL。 保留 message id、hash、offset、agent、session、cwd、sensitivity、tier。 不是默认注入 prompt 的记忆正文。 sources/conversations/*/messages.jsonl
MemoryItem 长期可复用的结构化结论。 多智能体共享、检索、治理、注入的核心单位。 不是 transcript，也不是 index row。 items/mem-*.md
Body MemoryItem 的 Markdown 正文。 detail 视图读取时使用。 不是 frontmatter 字段。 items/mem-*.md frontmatter 之后
ContextViews 同一个 item 的三档加载视图。 让注入从短到长渐进加载。 不是三份不同记忆。 context_views.locator/overview/detail_uri
Index 为了检索和排序建立的可重建投影。 提供 metadata 过滤、FTS、向量、graph、反馈统计。 不是事实源。 index.db
RetrievedItem 召回阶段的候选引用。 只告诉后续阶段 item id、score、BM25/向量排名和 trace。 不是完整 MemoryItem。 id, score, bm25_rank, vector_rank
ContextCandidate 防火墙评估用的候选。 把 MemoryItem、body、score、source、cluster_key 放在一起。 不是最终注入文本。 context_firewall_types.py
ContextPack 真正准备注入 prompt 的上下文包。 包含 selected_view、text、detail_uri、retrieve_hint、token 估算、CCR 信息。 不是新的记忆。 context_packing.py
Runtime Ledger 运行时事件账本。 记录 adapter、loop、feedback、recall gap、data flow。 不是长期结论本身。 runtime/*.jsonl
Review Candidate 待审批候选记忆。 隔离 proactive/semantic 生成内容，避免未验证内容污染事实源。 不是 MemoryItem。 review/proactive-candidates.jsonl
Pending 写入失败或 hook 兜底队列。 保证临时不可写时可以后续 sync-pending 重放。 不是写入成功。 pending/*.jsonl
最容易混淆的一句话：召回阶段返回的是 RetrievedItem，不是完整正文；分层上下文装载阶段才根据 id 读取 MemoryItem + body，再选择 locator、overview 或 detail。
02 / MemoryItem Anatomy

MemoryItem 里面每个字段承担什么责任。

这一层是 AMH 最核心的数据合同。字段不是装饰，后续的过滤、排序、衰减、防火墙、装载、治理都在读这些字段。

身份与范围

  • id：唯一 id，格式 mem-YYYYMMDD-HHMMSS-...。
  • type：fact、episode、decision、artifact、signal、handoff、policy、skill。
  • project / tenant_id：项目和租户过滤边界。
  • agent / session：谁写的，哪个会话写的。

检索与展示

  • title：人读标题，也是索引文本的一部分。
  • summary：短摘要，默认也会进入 locator。
  • tags：结构过滤、boost、adapter 证据识别。
  • context_views：locator、overview、detail_uri。

可信度与治理

  • confidence：默认 0.7，召回中直接乘进有效分。
  • retention：last_accessed、access_count、decay_class。
  • support_count / contradict_count / gain_score：反馈价值和治理信号。
  • maturity / abstraction：raw/L0、consolidated/L1、skill/L2。

来源与证据

  • refs.files：本地文件证据。
  • refs.urls：外部 URL 证据。
  • refs.commits：代码提交证据。
  • refs.resources / refs.extractions：资源和抽取 sidecar。
  • refs.mems：引用其他 memory。

时效与适用边界

  • validity.observed_at：何时观察到。
  • validity.ttl_hours：有效期。
  • validity.cwd/repo/branch/os/adapter：适用范围。
  • 防火墙会用这些字段判断 scope mismatch 和 stale state。

演化链

  • evolved_from：由哪些旧记忆演化而来。
  • superseded_by：被哪条新记忆替代。
  • version：版本号。
  • 召回和防火墙会阻断 superseded item，除非显式允许。
03 / Maintenance Chain

维护链路：把“发生过”变成“可信可复用”。

维护链路是 AMH 的前半条命。没有维护，召回只能把脏数据找得更快。维护负责判断什么能写、怎么写、怎么补证据、怎么修索引、怎么处理过期、重复、漂移和候选。

1维护信号用户写入、hook、transcript、resource、Loop、feedback。
2入口归一CLI/MCP/Web/SDK/hook 尽量收敛到统一写入语义。
3候选隔离proactive、semantic、pending、review 都不直接污染事实源。
4审计门critical/high 风险 fail-close。
5字段增强补默认 confidence、retention、context_views、decay_class。
6证据 sidecarresources/extractions/write ledger。
7事实源写入items/mem-*.md append 成功才算写入成功。
8索引投影items_meta、items_fts、items_vec、refs_graph。
9修复治理reindex、sync-pending、maturity、drift、summary rewrite。
10演化复核evolve/dream/proactive 批准后再回写。
维护链路逐环节解释
环节 概念 作用 为什么重要 边界
UserPromptSubmit 当前用户 prompt 的防丢证据。 进程异常或 adapter 没有 transcript 时，至少不丢用户输入。 保证原始输入可追溯。 不是 MemoryItem，不是完整会话。
Stop transcript 完整 transcript 导入。 把完整会话写到 sources/conversations。 这是原始对话权威证据。 不会默认注入 prompt。
write-memory / memory write 显式写长期记忆。 把用户或 Agent 提炼出的长期结论写成 MemoryItem。 长期复用必须靠提炼，不靠 transcript 垃圾堆。 应走 WriteService。
Harvest 从原始对话中抽取候选。 用 watermark 和 span hash 做断点续跑、去重。 避免重复归档同一段 transcript。 抽取候选仍不是长期事实。
Resource / Extraction 文件和抽取片段证据层。 保存资源 metadata 和抽取结果，可以后续 promote。 让文件证据可追溯。 资源本身不等于 MemoryItem。
WriteService 写入漏斗。 统一审计、字段增强、质量检查、证据 sidecar、写入账本、索引更新。 避免各入口乱写文件导致语义不一致。 索引失败不回滚 Markdown 事实源。
Index dirty 索引失败标记。 index/embedder 失败时标记后续修复。 保障“事实源写入”和“查询投影”解耦。 dirty 不等于写入失败。
Maturity governance 成熟度评分。 判断 item 是否仍是 raw，还是可以 consolidated/skill。 防止低证据内容被包装成高层知识。 是建议，不是随意自动改写。
Summary rewrite 摘要重写治理。 处理 summary 过长、影响检索和注入的问题。 摘要太长会污染 locator 和 FTS。 需要 snapshot / dry-run / rollback。
Auto governance 自动治理循环。 扫描 maturity、drift、conversation tier、index drift、evolve proposals。 让大脑不会无限积累噪声。 高风险动作进入 review_required 或 blocked。
04 / Recall Chain

召回链路：从用户问题到候选排序。

召回不是“搜一下向量库”。AMH 的召回分成结构过滤、FTS/BM25、向量检索、RRF 融合、候选策略阶段、过滤和扩展。它产出的仍是候选 item 引用，下一步才是防火墙和上下文装载。

1Query用户问题。
2SearchFiltertype、project、tags、tenant、since_days、superseded。
3Query ExpansionCJK/ASCII token、同义词、FTS5 安全转义。
4FTS/BM25SQLite FTS5 MATCH + bm25。
5Vectorembedding 检索；degraded 时跳过。
6RRF按排名倒数融合。
7Policy Stagesrerank、decay、feedback、boost、filter。
8MMR去重复，保多样性。
9Hopfield / Graph联想扩展、引用扩展。
10RetrievedItemid、score、rank、trace。
召回链路逐环节解释
环节 概念 输入 输出 意义
SearchFilter 结构过滤器。 type、project、tags、exclude_tags、since_days、tenant_id。 allowed_ids。 先缩小范围，避免语义相似但项目错误。
Query Expansion FTS 查询构造。 用户自然语言。 FTS5 安全 query。 把中文、英文、数字、下划线词、安全转义和同义词整理成可检索表达式。
FTS/BM25 全文检索排名。 title、locator、overview 的 FTS 索引文本。 BM25 hit list。 对路径、命令、错误码、函数名、adapter 名称最可靠。
Vector Search 语义向量检索。 query embedding。 vector hit list。 找表达不同但语义相近的记忆。
Degraded guard 退化保护。 embedder 状态。 BM25-only。 如果 embedder 是 degraded fallback，避免融合垃圾向量邻居。
RRF 倒数排名融合。 BM25 hits + vector hits。 初始候选分 S0。 不同检索器的原始分不可比，按排名融合更稳。
Cross-encoder rerank 可选重排。 query 和候选全文 pair。 sigmoid 后的相关性分。 在小候选池上用更贵模型做精排。
Decay 遗忘衰减。 confidence、decay_class、last_accessed、access/support/contradict/gain。 时间和反馈修正后的分。 旧状态自然降权，常用且被支持的记忆保活。
Feedback value 价值乘子。 support_count、contradict_count、gain_score。 bounded multiplier。 让真实采用过的记忆升温，被拒绝的降权。
Status / Runtime boost 场景化 boost。 query terms 和 item tags/type。 乘子提升。 handoff/status/runtime 相关问题优先拉真实状态证据。
Temporal / Supersession filter 旧状态过滤。 candidate metadata。 过滤后的 candidates。 避免过期状态和被替代记忆进入下一阶段。
MMR 最大边际相关性。 候选分和候选 embedding。 相关但不重复的候选列表。 避免上下文里出现一堆重复记忆。
Hopfield expand 联想式扩展。 当前候选 embedding 和分数。 围绕 attractor 的向量邻居。 补齐第一跳没搜到但与候选群相关的记忆。
Graph expand 引用图扩展。 top candidates 和 refs_graph。 邻居 item。 把相关证据、上下游决策和引用链带进候选池。
Trace / Access 召回轨迹和访问记录。 每阶段 before/after rank/score。 RetrievalTrace、access_count。 解释为什么这条 item 被选中，并给 retention 提供访问数据。
05 / Context Loading

分层上下文装载：从 item 候选到真正注入的文本。

召回拿到的是 item id 和分数。上下文装载才决定“给模型多少”。AMH 用 locator、overview、detail 和 detail_uri 做渐进式加载。

locator

最短定位视图，默认等于 summary。作用是让 Agent 知道“有这条记忆”，但不把正文塞进上下文。

低成本默认起点适合 brief

overview

结构化概览。适合 fact、decision、signal、handoff 或带 refs/validity 边界的 item。

中等成本解释边界可为空

detail

完整正文 body。只有显式请求、raw 直接证据、预算允许或强相关时才加载。可通过 detail_uri 或 retrieve_hint 再读。

高成本完整正文可逆读取
自动选择规则 
requested != "auto":
  直接尊重 locator / overview / detail

auto:
  默认 locator

  如果 item 是 raw + L0 + 有直接证据 refs + body 存在:
    升级 detail

  否则如果有 overview，并且满足以下任一条件:
    type 是 fact / decision
    type 是 signal / handoff
    有 refs 或 validity 边界
    firewall 决策是 demote
    升级 overview

预算不够:
  detail -> overview -> locator 逐级降级
关键语义：压缩视图不是原文。关键判断必须通过 detail_uri、MCP read_memory(... view='detail') 或 CLI memory read --view detail 读取 canonical body。
ContextPack 里包含什么
  • item_id：对应 MemoryItem。
  • selected_view：locator、overview 或 detail。
  • text：真正准备注入 prompt 的文本。
  • detail_uri：完整正文地址。
  • retrieve_hint：MCP 读取提示。
  • cli_retrieve_hint：CLI 读取提示。
  • packed_tokens / full_tokens：压缩前后 token 估算。
  • compressed / reversible：是否压缩，是否可逆追溯。
  • ccr_key / ccr_marker：压缩旁路信息。
06 / Context Firewall

上下文防火墙：把“搜得到”变成“能不能注入”。

防火墙不是召回算法，而是注入前策略门。它会对每个候选做 include、demote、exclude，再对整个 cohort 做覆盖率检查和预算约束。

规则 触发条件 结果 意义
sensitivity gate sensitivity 不在 public/internal。 exclude 默认不注入 private/secret。
superseded gate superseded_by 存在。 exclude 被新记忆替代的旧记忆不能继续误导。
review gate tags 含 needs-review / requires-review / review-rejected / unverified-boundary。 exclude 未复核边界不能进 prompt。
query mismatch query signal 可注入但候选不覆盖任何 query term。 exclude 避免无关记忆被带入。
very low confidence confidence < 0.2。 exclude 极低可信度直接阻断。
negative feedback contradict_count >= 3 且 gain_score <= -0.5 且 contradict 大于 support。 exclude 被多次否定的记忆不能再注入。
missing source fact/decision 缺 refs 来源。 exclude 事实和决策必须可追溯。
stale signal signal 创建超过 14 天。 exclude 临时信号不能长期误导。
stale handoff handoff 创建超过 30 天。 exclude 旧交接不能覆盖当前状态。
scope mismatch / stale temporal validity 与当前 cwd/repo/branch/os/adapter 不匹配或 TTL 过期。 exclude 带环境边界的状态必须按环境判断。
contested tags 含 contested / contradiction / conflict。 demote，分数乘 0.35 有争议内容降级，只给更少上下文。
low confidence confidence < 0.4。 demote，分数乘 0.5 低可信但未到阻断阈值。
L0 evidence only L0 且缺直接证据和 validity 边界。 demote，分数乘 0.2 原始低层抽象且证据弱，不应详细注入。
duplicate cluster 同 cluster 超过 max_per_duplicate_cluster=1。 保留最高，其他 exclude 防止重复内容占满上下文。
cohort strong anchor strong_terms 覆盖率低于 1.0。 reject cohort 如果整体候选没有覆盖强锚点，就整组拒绝。
07 / Algorithms

所有关键算法和公式。

这里严格区分“源码里有明确公式”和“调用底层库完成”。比如 BM25 由 SQLite FTS5 计算，AMH 自己只负责查询构造、结果取负和后续融合。

FTS / BM25

概念：FTS 是 Full Text Search，全文检索。AMH 用 SQLite FTS5 的虚拟表 items_fts，字段是 id, title, summary, body。

写入索引时：AMH 把 CJK 字符前后插空格，让 unicode61 tokenizer 能发出单字 token。写入 FTS 的文本是 title、context_views.locator、context_views.overview，展示时仍从 Markdown 事实源读。

查询时：用户 query 会被切成 ASCII/CJK token，token 会被引号包裹以防 FTS5 语法注入，同义词会扩展，默认用 OR 连接。

AMH 实现口径 
SELECT id, bm25(items_fts) AS score
FROM items_fts
WHERE items_fts MATCH ?
ORDER BY score
LIMIT ?

# SQLite FTS5 bm25(): lower is better
# AMH returns: Hit(score = -sqlite_bm25_score)
# 后续统一成 higher is better

意义：FTS/BM25 负责精确词命中，尤其适合路径、命令、错误码、项目名、adapter 名称。

Vector Search

概念：向量检索把 query 和 item 表示成 embedding，查语义邻居。

作用：补足 FTS 找不到的“表达不同但意思相近”的记忆。

边界：如果 embedder 处于 degraded fallback，AMH 会跳过 vector hits，走 BM25-only，避免把无意义向量邻居融合进来。

RRF 倒数排名融合

概念：RRF 解决 BM25 分数和向量分数不可直接比较的问题。它不看原始分，只看排名。

源码公式 
# rank0 从 0 开始
score = weight / (rrf_k + rank0 + 1)

# 用 rank1 理解
score = weight / (rrf_k + rank1)

S0 =
  bm25_weight  / (rrf_k + bm25_rank1)
+ vector_weight / (rrf_k + vector_rank1)

默认：rrf_k=60，bm25_weight=1.0，vector_weight=1.0。

例子：BM25 第 1，向量第 3：

S0 = 1/(60+1) + 1/(60+3)
   = 0.01639 + 0.01587
   = 0.03226
Cross-encoder rerank

概念：rerank 是精排。先用 BM25/vector/RRF 找候选，再把 top candidates 交给 cross-encoder 判断 query 和 item text 的匹配度。

模型：cross-encoder/ms-marco-MiniLM-L-6-v2。

启用：RERANK_ENABLED=1/true/yes。

得分归一化 
rerank_score = sigmoid(logit)
             = 1 / (1 + exp(-logit))

边界：这是可选增强，不是默认不可缺少的核心链路。

Confidence

概念：MemoryItem 的可信度字段，范围 0 到 1，默认 0.7。

更新规则 
confirm        -> confidence = 0.9
on_access      -> confidence += 0.01, cap = 0.95
on_reaffirm    -> support_count += 1
                -> gain_score += 0.1
                -> confidence += 0.02, cap = 0.95
on_reject      -> contradict_count += 1
                -> gain_score -= 0.2
                -> confidence -= 0.15
contradiction  -> confidence -= 0.15
supersede      -> confidence -= 0.15

召回中：confidence 会乘进有效分。

Decay 遗忘衰减

概念：decay 让旧记忆自然降权，但允许常用、被支持、有收益的记忆保活。

基础半衰期 
retention = 0.5 ** (days_since_reference / half_life)
decay_class half_life type 映射
architecture180 天artifact、skill
decision90 天decision、policy
fact60 天fact
episode30 天episode
ephemeral7 天signal、handoff
综合衰减系数 
access_multiplier =
  1 + min(0.35, log1p(access_count) * 0.08)

support_multiplier =
  1 + min(0.18, support_count * 0.03)

gain_multiplier =
  1 + clamp(gain_score * 0.12, -0.15, 0.15)

contradiction_multiplier =
  1 - min(0.45, contradict_count * 0.08)

decay_coefficient =
  retention
* access_multiplier
* support_multiplier
* gain_multiplier
* contradiction_multiplier

decay_coefficient bounded to [0.01, 1.35]
召回有效分 
effective = candidate.score * confidence * decay_coefficient
Feedback value

概念：feedback value 是召回排序中的“历史使用价值”乘子。

反馈写回 
adopted:
  support_count += 1
  gain_score += 0.1
  confidence += 0.02

rejected:
  contradict_count += 1
  gain_score -= 0.2
  confidence -= 0.15

ignored:
  no change
召回乘子 
raw =
  1.0
+ support_count * 0.03
- contradict_count * 0.10
+ gain_score * 0.50

feedback_value_multiplier = clamp(raw, 0.25, 2.0)

score = score * feedback_value_multiplier
Maturity 成熟度

概念：maturity 是治理和装载信号，不是当前召回主链里的直接乘法项。它判断一个 item 是否从 raw/L0 成熟到 consolidated/L1 或 skill/L2。

成熟度评分 
maturity_score =
  0.28 * source_completeness
+ 0.22 * confidence
+ min(0.18, support_count * 0.03)
+ min(0.12, access_count * 0.02)
+ min(0.12, len(refs.mems) * 0.04)
+ validation_evidence_score
+ 0.08 if overview exists
+ min(0.10, gain_score * 0.10)
- min(0.35, contradict_count * 0.07)
- 0.18 if stale/superseded
source_completeness:
  1.00 = has direct refs: files/urls/commits/resources/extractions
  0.55 = has refs.mems
  0.00 = no refs

validation_evidence_score:
  0.07 = tests/eval/benchmark/source evidence path
  0.04 = commit refs
  0.00 = none

classification:
  score >= 0.80 and (type=skill or abstraction=L2) -> skill / L2
  score >= 0.65 -> consolidated / L1
  else -> raw / L0
MMR 多样性重排

概念：MMR 避免一堆相似 item 同时进入候选前列。

公式 
MMR(candidate) =
  lambda * relevance
- (1 - lambda) * max_similarity_to_selected

relevance = candidate.score / max_score
similarity = cosine(candidate_embedding, selected_embedding)

lambda 越接近 1 越重视相关性，越接近 0 越重视多样性。

Status / Runtime boost、Graph、Hopfield
status/handoff boost 
当 query 是 stale/outdated/reorient/handoff + memory/project/status/trust/current:
  type == signal -> multiplier += 0.25
  tags 有 handoff/status/current/next-step -> multiplier += 0.15
adapter runtime evidence boost 
当 query 提到 runtime/hook/observed 且命中 adapter tag:
  adapter tag + runtime-evidence -> multiplier += 0.75
  hooks 或 real-config tag       -> multiplier += 0.25
  type == fact                  -> multiplier += 0.15
graph expansion 
neighbor_score = candidates[-1].score * 0.5
Hopfield expansion 
seed_weight = softmax(candidate.score)
query = weighted_sum(seed_embeddings, seed_weight)
attractor = HopfieldMemory.recall(query)
similarity = 1 / (1 + distance)
expanded_score = max_score * 0.85 * similarity
08 / Feedback & Governance

反馈不是点赞，而是改变未来召回和治理的信号。

AMH 只把明确 adopted/rejected 当成强反馈。retrieved 不等于 adopted，injected 不等于 adopted，ignored 不升温。

正确反馈

  • 这条记忆被 Agent 明确采用：adopted。
  • 这条记忆导致错误或被用户否定：rejected。
  • 任务结果明确列出 adopted_ids / rejected_ids。
  • 隐式正反馈只有严格条件下才会应用：任务成功、高置信、只有一个 injected memory、信号在 allow-list。

错误理解

  • 搜出来了就说明有用。
  • 注入了就说明采用。
  • 没被提到就自动升温。
  • 多条 memory 同时注入后，只凭任务成功就全部加分。
反馈如何进入治理闭环
  1. 注入 cohort 记录了哪些 item 被注入。
  2. 用户或任务结果明确指出 adopted_ids / rejected_ids。
  3. InjectionFeedback 校验反馈 id 必须属于 injected cohort，且不能同时 adopted 和 rejected。
  4. adopted 调 on_reaffirm，更新 support、gain、confidence。
  5. rejected 调 on_reject，更新 contradict、gain、confidence。
  6. 后续召回通过 decay 和 feedback value 读取这些字段。
  7. 成熟度评分、漂移检测、演化建议也会读取这些字段。
09 / Loop Engineering

Loop 是复杂任务账本，不是召回排序算法。

Loop 记录一次复杂任务的目标、上下文、预算、检查点、验证计划、验证结果、产物和结果。它让“做了什么”和“凭什么说完成”可追踪、可交接、可复盘。

1create创建 goal、project、cwd、adapter、session、budget、verification_plan。
2runningstart 或第一次 checkpoint 后进入运行。
3checkpoint追加阶段进展和 artifact。
4verification记录验证计划和验证结果。
5complete必须有 completion evidence。
6fail记录失败原因和证据。
7data-flowloop event 进入运行观测。
8evolutionfailed loops 触发 inspect_failed_loops 建议。
9handoff下个 Agent 可按 loop 状态承接。
10memory有价值结论再沉淀为 artifact/episode/handoff。

LoopRun 字段

  • loop_id、created_at、updated_at、status
  • goal、trigger、project、cwd、adapter、session_id
  • budget、context
  • verification_plan、verification_results
  • checkpoints、artifacts、outcome
  • memory_candidates、sensitivity

状态机

created -> running / failed / cancelled
running -> blocked / failed / completed / cancelled
blocked -> running / failed / cancelled
failed  -> running
completed -> none
cancelled -> none

complete:
  requires verification evidence
Loop 的价值
  1. 防止假完成：complete 没有 evidence 会失败。
  2. 支持长任务承接：下个 Agent 能读 goal、checkpoint、artifact、verification。
  3. 把过程和结论分开：Loop 是 runtime 账本，MemoryItem 是长期结论。
  4. 给治理提供负反馈：failed loop 进入 evolution control，触发 review_required 建议。
  5. 让评估闭环有证据：任务成功或失败不再只有聊天口头描述。

11 / Common Confusions

最容易误解的地方。

这部分可以直接变成 README 的“不要误读”小节。它专门处理你刚才问到的那些概念混乱。

误解：召回出来就是完整记忆正文

实际：召回出来的是 RetrievedItem 候选引用，只有 id、score、rank、trace。正文要在 context loading 阶段按 id 读取。

误解：summary、overview、detail 是三条记忆

实际：它们是同一条 MemoryItem 的不同视图。locator 默认常来自 summary；detail 是 canonical body。

误解：向量检索就是核心

实际：向量只是候选来源之一。FTS/BM25、RRF、decay、feedback、firewall、context loading 共同决定最终注入。

误解：maturity 是召回分乘子

实际：当前 maturity 主要用于治理和装载策略，不是召回主链里的直接乘法项。

误解：注入了就代表被采用

实际：只有明确 adopted 才升温；rejected 降权；ignored 不改变。

误解：Loop 是 memory item

实际：Loop 是 runtime 任务账本。它可以成为写 episode/artifact/handoff 的证据，但本身不等于长期记忆。
