Agent Memory Hub / Data Flow Mechanism

数据流转链路明细

这不是一张抽象架构图，而是按源码入口梳理 AMH 从生命周期触发、数据治理、写入落盘、索引投影、读取检索、上下文注入，到治理自进化的完整底层机制。

核心判断

AMH 的真实主链路是“本地 Markdown 记忆池 + SQLite 派生索引 + runtime/evidence/review sidecar + adapter hook 注入”。其中 items/mem-*.md 是唯一写入完成的事实来源；检索、图谱、上下文包、资源引用和 Web 诊断都是围绕它生成的派生视图。

触发矩阵：什么时候维护记忆

AMH 不依赖“会话结束后再想起来整理”。它把记忆维护拆成生命周期触发、显式写入、维护命令和治理演化四类。不同触发点只做自己该做的事：有的注入纪律，有的检索上下文，有的只记 runtime 证据，有的才真正写 MemoryItem。

触发点 什么时候发生 源码入口 读 / 写行为 落到哪里
SessionStart Agent 会话启动，adapter hook 触发。 agent_runtime_kit/hooks/inject-discipline.sh 注入记忆纪律；读取最近非自动的 signal；记录 runtime event。 <system-context>、runtime/adapter-events.jsonl
UserPromptSubmit 用户每次提交 prompt 前。 agent_runtime_kit/hooks/inject-context.sh 抽取 query signal；按 prompt 长度动态 top_k；走 context firewall；可记录 injection cohort / recall gap。 <agent_brain>、runtime/injection-cohorts.jsonl、runtime/recall-gaps.jsonl
Stop Claude Code 等 adapter 的 LLM turn 结束时；不是严格会话结束。 agent_runtime_kit/hooks/session-end-signal.sh 每个 session 去重后只写一个低噪声 signal，说明 transcript 待后续 harvest；不读全文 transcript，不调 LLM。 items/mem-*.md 或 pending/*.jsonl；去重 flag 在 .session-flags/

写入漏斗：什么才算“记忆已经写入”

01

入口只表达写意图

hook、CLI、MCP、SDK、Web、Hermes/provider 都只是入口。它们不应该各自决定如何落盘，正确目标是汇入同一个 WriteService。

02

MemoryItem 合同先成型

写入前形成 type/title/summary/tags/project/agent/source/confidence/refs/context_views 等结构化元数据，正文仍保留 Markdown 可读性。

03

审计门先于落盘

WriteService.write() 调用内容审计；critical/high 默认 fail-close，除非显式 allow_unsafe=True。harvest/pending/remember 等边界不明来源会被打上 needs-review、unverified-boundary 并降低置信度。

04

Markdown 是唯一完成语义

ItemsStore.write() 成功创建 ~/.agent-memory-hub/items/mem-*.md，才代表“记忆已经写入”。索引失败不推翻这个事实。

05

索引是派生投影

index.db 更新 meta/FTS/vector/refs_graph。失败时写入结果降级为 indexed=false、degraded=[index]，并追加 .index-dirty，由 memory reindex 或 memory verify --repair 修复。

P0 治理含义 当前系统的设计目标已经明确：所有写入口收敛到 WriteService。但源码里仍存在部分 import/evolve/内部执行器直接调用 ItemsStore.write() 的路径。它们需要继续收敛，否则会绕过统一审计、边界标记和质量告警。

记忆本体模型：不是一堆文本，而是可治理对象

AMH 的每条记忆都不是“纯文本向量”。它首先是一个 MemoryItem：用类型、成熟度、抽象层、置信度、引用、来源、有效期、敏感级别和上下文视图共同描述。检索时这些字段职责不同：类型/项目/标签用于候选过滤，置信度和衰减用于排序，成熟度用于治理、分层和复核边界，上下文视图用于装载。

记忆类型 写的是什么

  • fact：客观事实、限制、配置、API 行为。
  • episode：一次有学习价值的过程和结果。
  • decision：非显然技术决策及代价。
  • artifact：PR、文档、工具、报告等产出物。
  • signal / handoff：开放状态、阻塞、承接信息。
  • policy / skill：由演化链路沉淀出的规则和能力。

成熟度与抽象层 有多可复用

  • L0/raw：刚捕获、证据贴近现场，默认更谨慎。
  • L1/consolidated：多条证据提炼后的稳定知识。
  • L2/skill：可复用流程、策略或能力模块。
  • 成熟度不是全文检索的替代品，也不是当前搜索排序乘子；它是召回后的治理、分层和复核元数据。

衰减类别 记多久

  • architecture：半衰期 180 天。
  • decision：半衰期 90 天。
  • fact：半衰期 60 天。
  • episode：半衰期 30 天。
  • ephemeral：半衰期 7 天，适合 signal/handoff。
type -> decay_class: artifact/skill = architecture; decision/policy = decision; fact = fact; episode = episode; signal/handoff = ephemeral

分层加载，不等于分层检索

locator 用来让检索和快速判断成本最低；overview 用于常规上下文；detail 指向完整 Markdown body。也就是说，召回可以先用轻量视图，真正需要时再深读原文，避免把全部历史一次性塞进上下文。

不同介质为什么分开

Markdown 保留可审计事实；SQLite 保证本地全文/向量/图检索效率；JSONL/JSON sidecar 记录运行时、候选、原始证据和压缩原文。分开后任何派生层坏掉都能从 Markdown 或证据层重建，而不是让一个黑盒数据库吞掉所有语义。

存储介质地图：哪些数据写到哪里

AMH 不是单库系统，而是本地优先的多介质系统：Markdown 负责真实记忆，SQLite 负责检索效率，JSONL/JSON sidecar 负责证据、运行时、候选和降级队列。这样做的好处是可追溯、可恢复、可离线运行；代价是必须明确“谁是事实源、谁是派生物”。

介质 角色 写入者 读取者 治理语义
~/.agent-memory-hub/items/mem-*.md MemoryItem 源数据：frontmatter + Markdown body。 WriteService / 少量待收敛内部路径。 ItemsStore、Retriever、Web、MCP、SDK。 唯一事实源；append-only id；可 frontmatter 更新；archived 默认不参与扫描。
~/.agent-memory-hub/index.db 派生检索投影：meta、FTS、vector、refs_graph。 写入后 best-effort upsert、reindex、verify --repair。 Retriever、图谱、Web 诊断。 可删除重建；不能代表事实完成；漂移由 verify/reindex 修复。

~/.agent-memory-hub/runtime/injection-cohorts.jsonl 注入批次和过滤原因。 search-memory / context firewall。 召回质量分析。 支撑“为什么这次注入/没注入”。

~/.agent-memory-hub/runtime/compression-cache/*.json 上下文压缩的 CCR 原文缓存。 adaptive_compression.py。 retrieve_compressed_original()。 当没有 detail_uri 时保证可逆恢复。

读取、加载、注入：记忆如何进入 Agent 上下文

读取入口 多个 API，同一语义

  • hook 入口：inject-context.sh 在 UserPromptSubmit 前调用 search-memory.sh。
  • CLI 入口：memory search、memory brief、memory read。
  • SDK 入口：search_items() 返回 context_pack、retrieval_trace、firewall、resource_context。
  • Web 入口：/api/search 可打开 trace/firewall/resources 诊断。

查询前置 Query Signal

  • 短 prompt 默认只取少量候选，长 prompt 扩大 top_k。
  • 如果 query 过弱或不适合注入，hook fail-closed，返回空对象而不是硬塞上下文。
  • 默认排除 session-end、auto-captured，除非用户明确在问会话、handoff 或自动记录。
01

检索候选

Retriever.search() 先应用 type/project/tag/sensitivity 等过滤，再进行 BM25、vector、RRF fusion；后续叠加 handoff 补充、rerank、时间衰减、价值权重、runtime/status boost、stale/supersession 过滤、MMR 和可选图扩展。

02

防火墙筛选

启用 context_firewall 时，系统先取 top_k * 3 候选，再按主题匹配、时间冲突、置信度、反馈、敏感级别、review 状态、token budget 和 cohort 规则筛选。

03

构造 ContextPack

build_context_pack() 选择 locator/overview/detail 视图，并保留 detail_uri、加载原因、压缩信息和可恢复提示。预算不足时可从 detail 降级到 overview/locator。

04

可逆压缩

detail 文本超预算时，adaptive_compression.py 按内容类型选择搜索结果、build log、git diff、JSON array 或 plain text 压缩策略。没有 detail_uri 时写 CCR cache，并附 <<amh-ccr:key>> marker。

05

注入边界声明

hook 注入的 <agent_brain> 明确说明：这些是检索候选，不是聊天历史；当前用户消息和实时工具证据优先；memory 可能过期，使用前要看 cwd、adapter、时间和证据。

记忆召回细节：它到底按什么排

AMH 不是“只按相似度”，也不是“只按成熟度”。源码里的召回是一个多阶段决策链：先用结构化过滤限定候选空间，再用全文和向量做第一跳召回，用 RRF 合并排名，然后用衰减、反馈、状态、运行时证据、去冗余和上下文防火墙做治理。

S0 结构过滤 type / project / tags / exclude_tags / sensitivity / tenant / superseded
S1 并行召回 BM25 全文检索 + embedding 向量近邻；CJK 文本走分词修正。
S2 RRF 融合 把全文排名和向量排名变成统一候选分，避免单一路径失真。
S3 治理重排 置信度、保留率、反馈收益、状态 boost、runtime evidence。
S4 扩展去冗余 可选 cross-encoder；MMR、Hopfield attractor、refs graph 在主排序后扩展。
S5 安全注入 ContextFirewall 处理时间冲突、低置信、review、预算和敏感级别。

基础融合公式

RRF(d) = Σ_s w_s / (k + rank_s(d) + 1), s ∈ {BM25, Vector}

scoring_model.py 和 retrieval_fusion.py 明确了 RRF：排名越靠前贡献越大；BM25 和向量可以各自给分，避免中文关键词命中和语义近邻互相覆盖。

最终治理分

S_effective = S_rrf × confidence × decay_coefficient

置信度、遗忘衰减、反馈、状态和运行证据不替代检索，而是在候选已经相关之后决定“能否靠前、是否降权”。成熟度由治理链路计算，作为分层和复核元数据，不是当前搜索排序乘子。

遗忘曲线

retention = 0.5 ^ (days / half_life)

MemoryItem 检索层按 decay class 做指数衰减；原始对话证据层还会叠加 access、importance、support、gain、contradiction、noise 来分配 hot/warm/cold/frozen。

Hopfield 关联记忆

attractor = Σ_i softmax(β · sim(q, p_i)) · p_i

当前实现是轻量 continuous Hopfield-style expansion：第一跳候选的 embedding 形成 attractor，再去向量索引找“被候选群共同吸引”的邻居。默认不强依赖神经运行时，也不让 raw transcript 直接进入扩展。

MMR 去冗余

MMR = λ · relevance - (1 - λ) · max_similarity

当候选都很像时，MMR 会牺牲一点相似度换覆盖面，避免上下文里出现五条意思接近的记忆。

一句话回答排序问题 类型、项目、标签、租户、时间和替代策略先决定“能不能进候选池”；BM25/向量/RRF 决定“是否相关”；置信度、遗忘衰减、反馈、状态和运行证据决定“是否值得靠前”；MMR/Hopfield/图谱负责主排序后的关联扩展和去冗余；ContextFirewall 最后决定“能不能进当前上下文”。

数据治理与自进化：系统如何变得更好

质量治理

GovernancePipeline 扫描 ItemsStore，检查重复、噪声、TTL、质量问题。signal 默认更短 TTL，过期或低质量条目进入 review，而不是直接污染召回。

安全自动化

AutoGovernanceCycle 把动作分成 safe_apply、review_required、blocked。高风险的 archive/delete/consolidate/supersede/skill synthesis 不会在 auto 模式直接执行。

自进化

EvolveEngine 能提出 consolidate、promote、archive、generate_skill、crystallize、synthesize_skill、version_up 等 proposal。真实执行前跑 audit gate；失败则阻断。

主动记忆

proactive_memory.py 先把候选写到 review/proactive-candidates.jsonl。只有 approve 后，候选才通过 WriteService 创建 MemoryItem。

召回反馈

注入 cohort、recall gap、task outcome feedback 让系统知道“哪些记忆被注入、哪些没被注入、结果好不好”。这为后续权重、过滤和治理提供证据。

可验证压缩

compression_gate.py 用 few-shot case 检查 must_keep、must_drop、压缩率、tokens saved、可逆性，避免压缩把关键证据删掉。

自进化不是“自动乱改记忆” 现在的机制更像一个本地记忆操作系统：运行时先收集证据，治理层提出候选和 proposal，审计层决定是否阻断，review 层决定是否进入真实 MemoryItem。真正的革命性在于把“AI 自己记住了什么、为什么记住、何时可注入、如何撤回”都变成可审计对象。

机器学习 / 深度学习各自做了什么

AMH 的原则是“模型增强召回和建议，不默认接管事实写入”。也就是说，ML/DL 是可解释链路里的一个信号源，而不是让模型黑盒决定共享大脑该记什么、忘什么、注入什么。

能力 当前作用 源码入口 默认边界
语义 embedding 把 locator + overview 编成向量，用于语义近邻召回；默认多语言 MiniLM，支持中文语义空间。 platform/embedding.py、memory/recall/embedding_text.py 模型不可用时退到 HashingEmbedder，核心读写和 BM25 不崩。
向量检索 sqlite-vec 存储派生 embedding，和 BM25 一起进入 RRF。 platform/indexing/vector_index.py、HubIndex.vector_search() 向量是派生索引，不是事实源；可重建。
Cross-encoder rerank 可选深度模型重排 top pool，把 query-document pair 重新打分。 memory/recall/retrieval_rerank.py 只有 RERANK_ENABLED=1 才启用；不是默认生产路径。
Hopfield-style 关联扩展 把第一跳候选 embedding 看成关联记忆，形成 attractor，再找邻近记忆。 hopfield_memory.py、retrieval_hopfield.py 可选扩展；raw conversation 不直接进入，避免噪声放大。
LLM enrichment / compression harvest/enrich 和治理压缩可以用 LLM 辅助提炼，但都有机械 fallback 或 gate。 evidence/harvest/enricher.py、governance/compressor.py 离线优先；不能绕过审计和 review。
ML/DL advisory gate 评估模型候选是否只能 advisory、能否进 experiment、是否必须 hold。 evaluation/ml_advisory_gate.py 即使分数提升，也不能自动提升为默认；必须过 retrieval、compression、privacy gate。

Headroom 借鉴效果

AMH 没有把 Headroom 的代理层照搬进核心，而是内化了更适合记忆链路的部分：内容类型识别、领域压缩策略、CCR 可逆恢复、压缩指标和 few-shot gate。当前 adaptive_compression.py 支持 search results、build log、git diff、JSON array、plain text；compression_gate.py 用 must_keep/must_drop 防止“压缩反而丢关键证据”。

竞品借鉴效果

AMH 吸收的是“机制”，不是复制表面功能：Mem0 式多信号检索和事实生成，Letta 式 memory blocks/archival 分层，LangMem 式 semantic/episodic/procedural 区分，Zep 式图谱和时间上下文。落到源码里分别对应 MemoryType/Abstraction、Resource sidecar、BM25+vector+graph、retrieval trace、ContextFirewall 和治理审计。

论文、竞品与工程引用：借鉴了什么，没有神化什么

来源 可借鉴机制 AMH 当前落点 边界
Hopfield / Modern Hopfield 关联记忆、attractor、attention-like softmax association。 ContinuousHopfieldMemory、可选 hopfield_expand。 只做候选扩展，不让神经网络替代审计和事实源。
Ebbinghaus forgetting curve 时间衰减和半衰期思想。 retention_factor()、raw conversation hot/warm/cold/frozen。 加入访问、支持、矛盾、收益、噪声，不是只按时间删除。
RRF 多个检索器的排名融合。 BM25 + vector 的 rrf_fusion 和 trace。 RRF 是第一阶段相关性，不等于最终注入许可。
MMR 相关性与多样性平衡。 retrieval_mmr.py 可选去冗余。 避免重复，不替代上下文安全判断。
Netflix Headroom 内容路由、上下文压缩、可恢复引用。 adaptive_compression.py + compression_gate.py。 没有照搬 proxy；AMH 只拿适合 memory context 的机制。
Mem0 / Letta / LangMem / Zep 事实抽取、多层记忆、长期归档、图谱/时间上下文。 MemoryItem + resource sidecar + graph + trace + governance。 AMH 保持本地 Markdown 事实源，不把全部数据交给远端服务。

失败与降级语义：出问题时系统怎么保守处理

场景 系统行为 用户应该如何理解 修复入口
审计命中 critical/high WriteService fail-close，不写入 MemoryItem。 危险内容不会进入共享大脑；需要显式人工判断。 调整内容、降敏、或极少数场景显式 allow_unsafe。
Python/core 在 hook 阶段不可用 write-memory.sh 写入 pending/*.jsonl。 写意图不会丢，但还不算 MemoryItem 完成。 memory sync-pending。
索引 upsert 失败 Markdown 已写；标记 degraded=[index] 并追加 .index-dirty。 事实源存在，检索可能暂时不完整。 memory reindex 或 memory verify --repair。
query signal 太弱 inject-context.sh fail-closed，不注入记忆。 避免把无关记忆塞进上下文，降低污染。 用户 prompt 更具体；或改 query signal 规则。
ContextFirewall 排除候选 记录 firewall decision / cohort reason。 不是“没数据”，可能是置信度、时间、敏感度、review 或主题不匹配。 Web/API 打开 include_trace、context_firewall、include_resources。
adapter install-ready 但未 verified 配置可能存在，但缺 runtime event 或真实 smoke 证据。 “可安装”不等于“真实生命周期已跑通”。 跑 adapter doctor/smoke，观察 runtime/adapter-events.jsonl。
import/evolve 内部路径直接写 ItemsStore 能产出 Markdown，但可能绕过统一审计/质量标记。 这是 P0 收敛对象，不能当作最终理想架构。 把路径迁到 WriteService 或封装审计一致的 internal writer。

端到端例子：把底层机制串起来

例子 A 用户发起一个新任务

  • UserPromptSubmit 触发。
  • inject-context.sh 抽 query signal，决定 top_k。
  • search-memory.sh 走 Retriever + ContextFirewall。
  • 可注入候选被包装成 <agent_brain>。
  • Agent 看到边界声明：memory 是候选，实时证据优先。

例子 B Agent 写一条决策

  • Agent 遇到非显然技术决策，调用 write-memory.sh --type decision。
  • shim 调 memory_cli write；失败则写 pending。
  • WriteService 审计、补质量字段、标边界。
  • ItemsStore.write() 创建 items/mem-*.md。
  • 索引 best-effort；失败进入 .index-dirty。

例子 C 上下文即将压缩

  • PreCompact 触发。
  • lifecycle-event.sh 记录 runtime event。
  • 每 session 去重写 compact boundary signal。
  • 下次 resume 时，brief/search 能看到压缩边界。

例子 D 系统发现可提炼经验

  • proactive_memory 扫描近期高信号项。
  • 候选写入 review/proactive-candidates.jsonl。
  • 用户或治理流程 approve 后，才经 WriteService 写入。
  • 后续检索通过 refs/source 追溯到来源记忆。
