Agent Memory Hub 不是记忆库，而是跨智能体可信上下文操作系统。

它把原始会话、长期记忆、资源证据、派生索引、召回排序、上下文装载、治理演化和多智能体适配放在同一套可解释链路里。 核心问题不是“能不能召回”，而是：哪些事实可以进入共享大脑，如何证明它们从哪里来，什么时候该被降权、合并、归档或阻断，以及最终以什么粒度进入模型上下文。

一句话讲清楚

AMH 在每个智能体和它的上下文窗口之间建立一层可信中间层： 原始证据不直接污染 prompt，长期结论通过写入漏斗沉淀；召回不只是向量搜索，而是结构过滤、混合排序、成熟度和置信度治理、图/联想扩展、注入防火墙和分层上下文装载。

01 / One Real Chain

先维护，再召回，最后才注入。

旧版容易把重点压成“检索召回”。真实系统的主链路更长：如果事实源、证据、候选隔离、索引修复、反馈治理没有讲清楚，召回图再漂亮也解释不了“到底怎么取”和“为什么可信”。

1证据进入用户 prompt、完整 transcript、文件资源、任务结果、adapter/runtime event。
2候选隔离live-prompt、conversation、proactive candidate、pending 与 review 不等于长期记忆。
3写入漏斗WriteService 做审计、字段增强、质量检查、证据 sidecar。
4事实落盘items/mem-*.md 才是长期事实源；写入账本记录 provenance。
5索引投影FTS、向量、metadata、refs graph 是可重建投影；失败只标 dirty。
6维护治理maturity、drift、TTL、duplicate、summary rewrite、conversation tier。
7混合召回metadata filter 后 BM25 与向量并行，RRF 融合，rerank/decay/feedback 加权。
8防火墙topic、time、source、sensitivity、cohort、budget、duplicate cluster gate。
9分层注入locator / overview / detail，必要时通过 detail_uri 或 CCR 可逆取全文。
这条链路的关键语义：AMH 不把“查到”直接等价为“注入”。召回只是候选生成；真正进入上下文之前，还要过上下文防火墙、预算分配和分层装载。
02 / Architecture

可信上下文操作系统的五层结构。

这不是“一个 Markdown 文件夹 + 一个向量库”。真实架构要把证据层、事实源、派生投影、运行观测、接入面分开，否则很容易把原始 transcript、候选和长期记忆混成一团。

事实源层 source of truth

  • items/mem-*.md：长期记忆唯一事实源，承载 type、summary、tags、confidence、validity、refs。
  • sources/writes：写入账本，记录 writer、agent、session、hash、来源路径和有效期。
  • sources/conversations：原始消息证据，按 conversation id 存 JSONL，不默认注入 prompt。

派生层 rebuildable views

  • index.db：metadata、FTS、embedding、refs_graph，可重建。
  • resources / extractions：文件证据和抽取 sidecar，晋级时仍走 WriteService。
  • derived / runtime：层级摘要、CCR、召回缺口、注入 cohort、loop ledger。

控制层 policy gates

  • 写入前：audit gate、质量检查、候选隔离、敏感度和来源边界。
  • 召回中：结构过滤、RRF、rerank、decay、feedback、MMR/Hopfield/graph。
  • 注入前：Context Firewall、预算、分层装载、review / blocked / safe-apply。
03 / Maintenance First

维护链路：让大脑先可信，再让模型去取。

维护链路解决的是“什么能写、写到哪里、坏了怎么修、过期怎么降权、候选如何转正、哪些动作必须复核”。这比召回更基础，因为召回只会放大事实源的质量。

维护能力 它解决的问题 真实实现入口 不会越界做什么
写入漏斗 避免不同入口各自写文件，导致审计、证据、索引和账本不一致。 agent_brain/memory/store/write_service.py 不会把 index upsert 成功当作写入成功；Markdown 落盘才算事实源写入。
原始对话证据 保留可追溯事实来源，同时不把完整 transcript 默认注入 prompt。 conversation_store.py、memory conversation ingest/read/rebalance 不会把原始消息直接伪装成长期记忆。
资源与抽取晋级 文件、图片、抽取片段可以作为证据进入 sidecar，并在确认后晋级成 MemoryItem。 resource_store.py、extraction_promotion.py、memory resource promote-extraction 不会把多模态 placeholder 当作真实文本证据。
索引修复 embedding 或 index 失败不能丢掉已经写入的长期事实。 memory reindex、memory verify --repair、memory sync-pending 不会因为向量服务失败回滚 Markdown 写入。
自动治理 成熟度、过期、漂移、重复、索引偏差、conversation tier 需要持续扫描。 auto_governance.py、maintenance_plan.py、memory govern auto/plan 高风险合并、归档、替代、删除不会默认自动执行。
摘要重写 长摘要会污染检索和注入，需要有快照、dry-run、rollback 机制。 summary_rewrite_apply.py、memory govern apply-summary-rewrites 不会在没有历史快照和回滚标记的情况下静默改写。
Loop ledger 复杂任务要留下 goal、budget、checkpoint、verification、artifact 和 outcome。 memory loop create/checkpoint/complete/fail、runtime/loops/*.json 不会在缺少完成证据时把 loop 标记完成。
04 / Recall Sequence

召回链路：从用户问题到分层上下文注入。

真实读取顺序不是一句“召回”。它先做结构过滤，再并行跑全文和向量；融合后继续经历重排、成熟度、置信度、遗忘衰减、图扩展、联想扩展、上下文防火墙，最后才决定 locator、overview 还是 detail。

顺序 步骤 做什么 关键代码 / 对象
1 问题解析 分析 query signal、项目/类型/标签/时效等约束，形成结构化 SearchFilter。 SearchFilter、analyze_injection_query
2 元数据过滤 先按 type、project、tenant、tags、exclude_tags、since_days、superseded/stale 状态收窄候选池。 retrieval.py metadata filters
3 全文与向量并行 FTS/BM25 与 embedding 检索并行；embedder 不可用时退化为 BM25-only，不让召回链路中断。 HubIndex、FTS、embedding backend
4 RRF 融合 按倒数排名融合全文和向量结果：score += weight / (rrf_k + rank + 1)。 retrieval_fusion.py
5 重排与有效分 可选 cross-encoder rerank；再合入 confidence、遗忘衰减、反馈 value、runtime evidence、handoff/status boost。 retrieval_decay.py、rerank、feedback stats
6 时态与替代过滤 过滤 stale temporal state、superseded item，避免旧状态压过当前证据。 stale temporal filter、supersession filter
7 扩展与多样性 MMR 降低重复，Hopfield 做联想补全，refs_graph 做关联扩展。 MMR、Hopfield、refs_graph
8 注入防火墙 按来源、敏感度、review 状态、查询匹配、时间冲突、topic recency、duplicate cluster、cohort 和预算做 include/demote/exclude。 context_firewall.py
9 上下文装载 默认 locator；需要时 overview；只有直接证据、强相关、预算允许时升 detail。detail 可通过 detail_uri / CCR 可逆读取。 context_loading.py、context_packing.py
10 反馈回写 明确 adopted/rejected 才改 support、contradict、gain、confidence；ignored 不升温。 injection-feedback、runtime cohorts

06 / Algorithm Detail

算法不是“加个向量库”，而是多阶段质量控制。

AMH 的检索质量不是靠某一个模型能力，而是靠多个保守、可解释、可退化的阶段叠加：结构过滤负责范围，BM25/向量负责候选，RRF 负责融合，衰减和反馈负责时间与价值，防火墙负责最后的上下文安全。

算法 / 机制 输入 输出 为什么重要 真实边界
SearchFilter query、type、project、tags、tenant、since_days、stale/superseded 策略 收窄后的候选池 先限制范围，减少“语义相似但项目错误”的召回。 过滤条件过窄会漏召回，需要 brief/detail 读取补证据。
Query expansion / CJK tokens 用户问题、关键词、同义词、中文分词信号 全文检索查询变体 中英混合、项目名、缩写、中文短词都不能只靠空格切词。 扩展是召回辅助，不是事实创造。
BM25 / FTS 正文、summary、tags、metadata 文本 按关键词相关度排序的候选 对精确路径、命令、错误码、文件名比向量更可靠。 语义泛化弱，需与向量融合。
Vector retrieval embedding query 与 item embedding 语义相近候选 找得到换了说法的决策、经历和交接信号。 embedder 失败时退化，不应阻断核心检索。
RRF BM25 rank、vector rank、权重、rrf_k 融合分 不同检索器分数不可直接比较，排名融合更稳。 RRF 只能融合候选，不能判断事实是否仍有效。
Cross-encoder rerank query + candidate pair 更细粒度相关性排序 在候选集较小后用更贵模型重排，提升精度。 可选增强，不能成为默认不可用时的单点故障。
Retention decay 年龄、half-life、access/support/gain、contradiction、confidence effective score 旧状态必须自然降权；常用、被支持、有收益的记忆可以保持活性。 衰减不是删除，删除/归档仍走治理。
Feedback value adopted/rejected 注入反馈、task outcome support、contradict、gain、confidence 调整 让“被模型真实采用且带来结果”的记忆升温，错误注入降权。 ignored 不应被解读为采用。
MMR 候选相关度与候选间相似度 相关但不重复的列表 避免上下文里塞进十条同义记忆。 多样性不能压过强证据。
Hopfield / graph expansion 候选、关联边、引用图、联想状态 补充关联记忆 让交接、决策、文件证据和后续结果能串起来。 扩展结果仍要过防火墙和预算。
Context Firewall 候选、query signal、预算、source/sensitivity、temporal/topic rules include / demote / exclude 决策与原因 把“检索相关”升级成“可安全注入”。 防火墙是注入前门，不替代写入审计。
Context loading 防火墙后的候选、预算、证据强度 locator、overview、detail、detail_uri、CCR 上下文预算有限，必须按任务需要逐级展开。 压缩视图不能当完整原文，关键判断要读 detail。

10 / Hard Problems

真正难点：不是“存下来”，而是长期不污染、不漂移、不越权。

AMH 做了很多看起来不像“功能亮点”的工程，它们才是开源读者会关心的硬问题：失败时怎么降级、错误记忆怎么不被放大、候选怎么不误入事实源、多个智能体怎么不互相污染。

难点 为什么难 AMH 的处理方式 代码入口
写入与索引一致性 索引、embedding、FTS 都可能失败；如果失败时丢写入，会损坏用户信任。 Markdown 写入是事实源；索引失败只标 dirty，后续 repair/reindex。 write_service.py、verify、reindex
原始 transcript 污染 完整对话噪声大、敏感、上下文占用高，不能直接当长期知识。 ConversationStore 存证据；MemoryItem 只存提炼后的长期结论。 conversation_store.py
旧状态压过新证据 Agent 最容易被过期 handoff、旧路径、旧命令误导。 stale temporal filter、supersession、decay、topic recency、Context Firewall。 retrieval.py、context_firewall.py
上下文预算爆炸 承接任务时 bulk-read 很容易把窗口吃满，模型反而无法行动。 brief/locator/overview/detail 分层；detail_uri、head 限制、CCR 可逆压缩。 context_loading.py、context_packing.py
候选自动晋级风险 proactive/semantic 候选可能合理但未验证，不能直接成为事实。 review queue 隔离；approve 后仍走 WriteService。 proactive_memory.py
多智能体能力虚标 模板可安装不等于真实客户端能工作，更不等于产生运行证据。 adapter truth contract：doctor、install-verify、runtime_observed、uninstall-check 分级。 adapter CLI、Web adapter APIs
治理误伤事实源 自动合并、归档、删除如果做错，会永久破坏共享大脑。 maintenance plan 分 safe/review/blocked；高风险动作必须复核。 maintenance_plan.py、auto_governance.py
增强模型单点依赖 cross-encoder、Headroom、ML advisory 可提升质量，也可能不可用或不稳定。 作为 gate/advisory/optional bridge；失败时有本地 fallback 或退化路径。 headroom、compression-gate、ml-advisory-gate
Web/API surface 漂移 路由拆分和重构容易漏端点、重复端点或丢 internal symbol。 conformance test 锁定 89 routes 和 MCP 27+1 surface。 test_web_surface_lock.py、test_public_surface_lock.py
Loop 完成证据 复杂任务如果只记录“完成了”，下个 Agent 无法验证真假。 Loop complete 要求 completion evidence，事件账本记录 checkpoint、verification、artifact。 loop_store.py、loop_events.py
11 / Truth Boundaries

专业 README 也要敢写边界。

开源前最需要的不是把话说满，而是让读者相信你知道哪些地方已经验证、哪些地方是保守门禁、哪些地方不能宣传成默认能力。

不能说：“所有 Agent 都已经真实端到端验证。”
当前 adapter 账本是 16 个，其中 7 verified、8 install-ready、1 wip，runtime observed 为 3。
不能说：“AMH 自动维护所有 memory，不需要人管。”
它会自动扫描和 safe-apply 低风险动作，但高风险合并、归档、替代、删除和技能合成必须走 review/evolve。
不能说：“向量召回就是核心能力。”
向量只是候选来源之一；metadata、BM25、RRF、decay、feedback、firewall、context loading 才构成完整读取链路。
不能说：“原始对话就是长期记忆。”
原始对话是证据层；长期记忆必须提炼成 MemoryItem，并保留 provenance。
不能说：“ML/DL 已接管默认治理。”
ML advisory、compression gate、retrieval gate 是建议和质量门禁，不是默认事实源控制器。
不能说：“所有派生视图都等价于事实。”
index.db、hierarchy、runtime、review、derived 都是投影、候选或观测，不能反向冒充 facts。
