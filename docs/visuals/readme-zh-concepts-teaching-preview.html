00 / Reader Route

读者应该先读哪条线。

旧版难懂的原因是太像源码索引：一上来就把 FTS、RRF、MMR、Hopfield、confidence、maturity 摊开，读者还没建立对象模型。新版先用一个真实工作故事建立“什么东西在哪一层”，再进入公式。

教学优先，不牺牲完整性
第一遍：读故事

只看“一条记忆的完整故事”和“五个对象”。先知道 item、body、context pack、loop 各自是什么。

第二遍：读两条链路

维护链路解释“怎么写、怎么变可靠”；召回链路解释“到底怎么取”。顺序不能反。

第三遍：查公式

RRF、decay、feedback、MMR、maturity 等公式放在算法区，避免把新读者挡在门外。

第四遍：看 Loop

Loop 单独成章，因为它横跨任务过程、治理、演进和记忆沉淀，但不直接替代召回算法。

01 / One Memory Story

先用这次 README 任务走一遍完整系统。

下面这个例子故意不用抽象名词开场。它从“用户要求把概念页讲得事无巨细”开始，一直走到未来另一个 Agent 搜“Loop 到底用在哪里”时，AMH 如何把正确上下文注入回来。

贯穿例子
Current User Prompt

用户说了什么

“不不不，你还是跟之前一样，详细解释清晰，可以在 html 中预览出来给我看。但一定要事无巨细，每个环节、每个链路、每个名词、每个算法。”

这句话本身是当前会话事实。它不应该直接变成长久记忆，除非我们从中提炼出对未来有价值的 artifact、decision、handoff 或 fact。

系统真正要做的分层

  1. 当前 prompt 先成为运行时输入，必要时还会被 live-prompt 兜底保存，避免异常丢证据。
  2. 完成 HTML 产物后，如果它对未来有复用价值，应该写一条 artifact MemoryItem，正文说明产物路径和用途。
  3. MemoryItem 写入后，索引层派生出 metadata、FTS、vector、refs_graph 等检索材料。
  4. 未来查询“Loop 用在哪里”时，先过滤候选范围，再并行跑全文和向量，RRF 融合，候选排序与扩展。
  5. 注入前防火墙判断是否过期、低置信、缺来源、敏感或主题不匹配。
  6. 最终 ContextPack 只装载必要层级：locator、overview 或 detail；detail 通过 detail_uri 可渐进读取。
01

当前用户问题进入 Agent 运行时。

当前 prompt 是本轮任务输入。UserPromptSubmit hook 可以写 live-prompt 作为防丢证据；Stop hook 如果拿到 transcript_path，导入完整 transcript，这是原始会话的权威证据源。

02

是否写长期记忆，需要触发器。

不是每句话都写。只有 decision、fact、signal、episode、artifact、handoff 这类对未来有复用价值的内容，才应该写入 MemoryItem。

03

HTML 产物完成后，写 artifact MemoryItem。

正文必须包含 产出物 和 用途。这条 item 不是 HTML 源码本身，而是未来能搜到“有这么个产物、在哪、何时用”的结构化线索。

04

MemoryItem 变成长期事实源。

item 保存 title、summary、refs、confidence、retention、context_views、source、validity、support_count、contradict_count、gain_score 等字段。

05

索引层从 item 派生检索视图。

metadata 支持结构过滤；FTS5 支持全文/BM25；vector 支持语义近邻；refs_graph 支持引用邻居扩展。索引是派生物，不是权威正文。

06

未来查询先过滤，再并行检索。

项目、类型、标签、租户、时间等先缩小候选池；随后 BM25/FTS 和 vector 并行取 top 候选。

07

RRF 融合排名，候选进入排序阶段。

RRF 只看各通道排名，不强行比较 BM25 原始分和向量距离。可选 rerank、decay、feedback value、status/runtime boost、supersession filter、MMR、Hopfield、graph expansion 逐步处理。

08

防火墙决定“能不能进当前上下文”。

召回相关不代表可以注入。敏感级别、缺来源、过期 handoff/signal、低 confidence、负反馈、review 标记、主题覆盖不足都会影响 include/demote/exclude。

09

ContextPack 决定给模型多少信息。

默认从 locator 开始。需要更多时读取 overview；关键判断或原始/L0 证据需要 detail 时，再通过 detail_uri 有界读取正文。

10

反馈改变未来，不代表这次一定写新记忆。

adopted/rejected 会改变 support_count、contradict_count、gain_score、confidence；这些字段会被后续 decay 和 feedback value 读取。

11

Loop 记录复杂任务过程和验证证据。

LoopRun 保存目标、预算、上下文、检查点、验证计划、验证结果、产物和 outcome。它能被治理和演进读取，但不会自动变成 MemoryItem，也不是召回排序算法。

02 / Object Model

先把五个对象分清，后面的公式才有意义。

很多误解来自把 evidence、item、retrieved item、context pack 和 loop 混成一个东西。AMH 的工程复杂度正是来自它们被刻意分层。

对象边界
Evidence

原始证据

用户 prompt、完整 transcript、文件、URL、commit、resource、运行事件、loop event。它回答“事实从哪里来”。

不是：默认注入 prompt 的长期记忆。

MemoryItem

长期记忆条目

经提炼的 fact、decision、artifact、handoff 等。它有 title、summary、refs、confidence、context_views。

不是：完整聊天记录或代码仓库。

RetrievedItem

召回候选

搜索阶段产出的候选引用，包含 id、score、bm25_rank、vector_rank、trace。

不是：已经要注入的完整正文。

ContextPack

上下文包

注入前的最终文本载体，选择 locator、overview 或 detail，并带 detail_uri、token 估算和读取提示。

不是：无限制全文塞入。

LoopRun

任务账本

记录复杂任务目标、检查点、验证、产物、结果和 memory candidates。

不是：RRF、decay、MMR 的打分项。

MemoryItem 字段逐个解释
字段作用读者应该怎么理解
id全局条目标识。召回、反馈、读取正文都靠它定位。
typefact、episode、decision、artifact、signal、handoff、policy、skill。决定治理语义和默认衰减类别。
title / summary人可读标题和短摘要。用于列表展示、locator、索引和快速判断。
refs文件、URL、commit、resource、其他 memory 的来源引用。回答“凭什么相信”。fact/decision 缺来源会被防火墙排除。
confidence0 到 1，默认 0.7。召回有效分会乘它；反馈会轻微上调或下调。
retention访问次数、最后访问时间、保留策略。遗忘衰减和访问加成读取这里。
abstraction / maturity抽象层级与成熟度。治理和装载信号，当前不是召回主链直接乘法项。
context_viewslocator、overview、detail_uri。实现渐进式上下文加载；先短后长。
source / validity来源类型、有效期和适用边界。帮助防火墙处理过期、时间范围和上下文不匹配。
support_count / contradict_count / gain_score反馈统计。后续 feedback value、decay、maturity 会读取。
superseded_by被新条目替代。召回和防火墙默认会阻断被替代的旧 item。

最关键的一句话

分层上下文装载之前，召回拿到的是 RetrievedItem 候选，也就是 item 的引用、分数和轨迹；到了装载阶段，系统才基于 item 的 context_views.locator、context_views.overview 和 detail_uri 决定给模型多少内容。

03 / Maintenance Pipeline

维护链路要先讲：它决定记忆是否值得被未来召回。

如果只讲召回，读者会误以为 AMH 只是“从向量库找相似文本”。真正的难点在前半段：什么能写、怎么证据化、怎么索引、怎么治理、怎么处理过期和冲突。

先维护，再召回
输入/证据-> 写入触发器-> WriteService-> MemoryItem-> 索引投影-> 治理评分-> 反馈修正-> 演进/复盘
维护链路逐环节说明
环节输入输出为什么重要
原始证据采集 prompt、transcript、文件、resource、adapter runtime event、loop event。 证据层记录。 保留事实来源，避免 MemoryItem 变成无出处结论。
写入触发器 工作中出现非显然决策、客观事实、未解决 signal、有学习价值 episode、产物 artifact、handoff。 一条候选长期记忆。 质量大于数量，阻止把临时噪声写进共享大脑。
WriteService / CLI type、title、summary、tags、project、agent、session、正文。 结构化 MemoryItem 文件。 统一 schema、正文格式、敏感级别和来源字段。
ContextViews 生成 summary、overview、body/detail_uri。 locator / overview / detail_uri。 未来注入时可以渐进加载，不必一次塞全文。
索引投影 MemoryItem 与正文。 items_meta、items_fts、items_vec、refs_graph。 把权威 item 投影成可搜索结构；索引可重建，不是权威事实源。
成熟度评分 来源完整度、confidence、support、access、refs、验证证据、overview、gain、contradict、stale。 raw/L0、consolidated/L1、skill/L2。 帮助治理、审查和分层装载，但当前不是主召回公式的直接乘子。
遗忘与保留治理 MemoryType -> DecayClass -> half life，last_accessed_at，access_count。 retention factor 与治理建议。 让短期 handoff/signal 更容易过期，架构/skill 保留更久。

反馈回写 adopted_ids、rejected_ids、任务 outcome。 support_count、contradict_count、gain_score、confidence 更新。 让未来排序读取真实使用效果，而不是只靠初始相似度。
Loop / Evolution LoopRun、loop event、失败事件、验证结果、产物。 治理输入、演进信号、可能的 memory candidate。 把过程证据和长期结论分开；复杂任务可复盘、可交接。
04 / Recall Pipeline

召回链路回答“到底怎么取”。

用户问题进来后，系统不是只搜向量。真实读取顺序是：用户问题 -> 元数据/记忆类型过滤 -> 全文与向量并行 -> RRF -> rerank/成熟度信号/置信度/遗忘衰减/反馈 -> MMR/Hopfield/图扩展 -> 防火墙 -> 分层上下文注入。

召回时序
用户问题-> SearchFilter-> FTS/BM25+ Vector-> RRF-> rerank/decay/feedback-> MMR/Hopfield/Graph-> Firewall-> ContextPack
召回链路逐环节说明
顺序环节做什么输出
1用户问题解析拿到自然语言 query，并可结合 project、tenant、tags、type、时间范围等过滤条件。Query + SearchFilter。
2元数据/记忆类型过滤先用 metadata 缩小可搜索集合：项目、类型、标签、敏感级别、替代状态等。候选范围。
3AFTS/BM25在 FTS5 中搜索 title、locator、overview；CJK 会做分词处理。BM25 排名。
3B向量检索用 embedding 做语义近邻；如果 embedder 降级，则跳过向量，保留 BM25-only。vector 排名。
4RRF 融合把 BM25 和 vector 两个不可直接比较的排序融合。基础候选分。
5可选 rerank环境开关启用时，用 cross-encoder 对 query-item 文本相关性精排。重排后的候选。
6confidence / decay把置信度、遗忘衰减、访问加成、支持/反驳、gain 等乘入有效分。有效分。
7feedback value把历史采用、拒绝和任务收益转成乘子。价值修正分。
8状态/运行证据 boost对 stale/status/handoff 类查询或 adapter runtime evidence 查询做窄 boost。场景感知排序。
9temporal / supersession filter过滤时间状态不适配或已被替代的条目。更干净的候选。
10MMR减少多个相似 item 同时占满 top-K，保留多样性。去冗余候选。
11Hopfield / Graph expansion可选关联扩展；Hopfield 从第一跳候选形成 attractor，graph 从 refs_graph 找邻居。补充候选。
12Context Firewall注入前按敏感、来源、置信、过期、主题覆盖、负反馈等治理规则 include/demote/exclude。可注入候选。
13分层上下文装载按预算和 item 状态选择 locator、overview、detail。ContextPack。
05 / Injection Governance

召回相关，不代表可以注入。

这是 AMH 和普通 RAG 很不一样的地方：召回只是“可能相关”，注入要再过治理门。最后给模型的文本也不是全文优先，而是按信息密度和风险分层装载。

Firewall + ContextPack

Context Firewall 检查什么

  • 敏感级别是否允许：默认允许 public/internal。
  • 是否已 superseded，或处于 review/rejected 状态。
  • fact/decision 是否缺少来源。
  • confidence 是否过低，负反馈是否过强。
  • signal/handoff 是否过期。
  • 候选是否与当前 query 主题覆盖不足。
  • 同一 duplicate cluster 是否重复注入。

ContextPack 怎么装载

  • locator：最短定位信息，通常来自 summary。
  • overview：压缩概览，适合事实/决策/signal/handoff 或带来源边界的条目。
  • detail：正文内容；预算足够且关键判断需要时才读。
  • detail_uri：可逆读取路径，让 Agent 后续有界读取正文。
  • retrieve_hint：告诉 Agent 这只是压缩视图，关键判断要读取 detail。
分层装载的决策规则

自动模式从 locator 起步。若 item 是 raw/L0、直接证据 refs 较强且 body 存在，可能选择 detail；若 overview 存在且 item 是 fact/decision、signal/handoff、有 source refs、有 validity 边界或曾被 firewall demote，则优先 overview。预算不足时会从 detail 降级到 overview，再降级到 locator。

requested view explicitly set -> honor it
else auto:
  raw/L0 + direct evidence refs + body -> detail
  overview exists and governance needs context -> overview
  otherwise -> locator

budget pressure:
  detail -> overview -> locator
06 / Algorithms

所有关键算法和公式放在这里查。

这里不再用大词糊弄：每个算法都说明“它解决什么问题、输入是什么、输出是什么、当前代码怎么计算、它不负责什么”。

公式区
FTS / BM25

全文检索是什么

FTS 是 Full-Text Search，全称全文检索。AMH 使用 SQLite FTS5，对 id,title,summary,body 建全文索引，实际写入检索文本时使用 title、locator、overview 等压缩视图。

BM25 是 FTS5 返回的相关性排序。SQLite 的 bm25() 原始分越小越好；AMH 返回 hit 时取负数，让“大分更好”的语义统一。

RRF

倒数排名融合

RRF 解决“BM25 分数”和“向量距离/相似度”不能直接相加的问题。它只看排名，不看原始分。

score(item) += weight(channel) / (rrf_k + rank + 1)

当前默认 rrf_k=60，BM25 和 vector 权重默认都是 1.0，rank 是从 0 开始的排名。

Rerank

精排怎么来

rerank 是可选 cross-encoder 精排。先由 BM25/vector/RRF 找候选，再用 cross-encoder 评估 query 和 item text 的匹配度。

当前实现通过环境变量 RERANK_ENABLED=1/true/yes 启用，模型名默认 cross-encoder/ms-marco-MiniLM-L-6-v2，logit 经过 sigmoid 变成 0 到 1 的分数。

Confidence

置信度怎么变

confidence 是 MemoryItem 字段，范围 0 到 1，默认 0.7。召回有效分会乘它。

on_access:       +0.01, cap 0.95
on_reaffirm:    support += 1, gain += 0.1, confidence += 0.02
on_reject:      contradict += 1, gain -= 0.2, confidence -= 0.15
on_supersede:   confidence -= 0.15
Decay

遗忘衰减怎么计算

AMH 按 MemoryType 映射到 DecayClass。architecture 180 天半衰期，decision 90 天，fact 60 天，episode 30 天，ephemeral 7 天。

retention = 0.5 ** (days_since_access / half_life)
access    = 1 + min(0.35, log1p(access_count) * 0.08)
support   = 1 + min(0.18, support_count * 0.03)
gain      = 1 + clamp(gain_score * 0.12, -0.15, 0.15)
contra    = 1 - min(0.45, contradict_count * 0.08)
coefficient = clamp(retention * access * support * gain * contra, 0.01, 1.35)
effective_score = score * confidence * coefficient
Feedback Value

历史使用价值怎么进排序

feedback value 把历史采用/拒绝/收益转成召回乘子。

raw = 1
  + support_count * 0.03
  - contradict_count * 0.10
  + gain_score * 0.50

multiplier = clamp(raw, 0.25, 2.0)
score = score * multiplier
Maturity

成熟度不是召回主公式

maturity 判断 item 从 raw/L0 到 consolidated/L1 或 skill/L2 的成熟程度。它是治理和装载信号，当前不是召回主链的直接乘法项。

score =
  0.28 * source_completeness
  + 0.22 * confidence
  + min(0.18, support_count * 0.03)
  + min(0.12, access_count * 0.02)
  + min(0.12, len(refs.mems) * 0.04)
  + validation_evidence_score
  + 0.08 if overview exists
  + min(0.10, gain_score * 0.10) if gain > 0
  - min(0.35, contradict_count * 0.07)
  - 0.18 if stale or superseded
MMR

多样性重排

MMR 避免多个相似 item 占满 top-K。它在“相关性”和“与已选结果的相似度”之间做贪心平衡。

MMR(candidate) =
  lambda * relevance(candidate)
  - (1 - lambda) * max_similarity(candidate, selected)

relevance 来自候选分归一化；similarity 使用 embedding cosine。

Hopfield

关联扩展，不是默认主检索

Hopfield expansion 是可选的关联扩展。它用第一跳候选的分数 softmax 形成 weighted query，再调用 continuous Hopfield recall 得到 attractor，随后用 attractor 做向量搜索找“共同吸引”的邻居。

seed_weight = softmax(candidate_scores)
weighted_query = sum(seed_weight_i * embedding_i)
attractor = ContinuousHopfieldMemory.recall(weighted_query)
similarity = 1 / (1 + distance)
expanded_score = max_score * 0.85 * similarity
Graph Expansion

引用图扩展

Graph expansion 从 top hits 的 refs_graph 找邻居。它用于补充与已命中 item 有来源或引用关系的条目。

neighbor_score = tail_candidate_score * neighbor_score_factor

默认 neighbor_score_factor 为 0.5。它是补充扩展，不应该替代主排序。

Status / Runtime boost 的边界

Status boost 是窄场景规则：当 query 同时像是在问 stale/outdated/reorient/handoff，又和 memory/project/status/trust/current 等上下文相关时，signal/handoff 会被补充或加权。Runtime evidence boost 则针对 adapter runtime/hook/observed 类查询，对 adapter tag、runtime-evidence、hooks、real-config、fact 做加权。

这些 boost 的价值是让“当前状态、真实运行证据、适配器能力边界”不被普通语义相似度淹没。它们不是通用魔法权重。

07 / Feedback Loop

反馈不是点赞，而是改变未来召回和治理。

AMH 区分“注入过”和“被采用”。一个 item 被注入到上下文，不代表它对任务真的有帮助。只有明确 adopted 或 rejected，才应该写入强化或惩罚。

Adoption-aware

adopted

代表该记忆被本轮任务实际采用。会增加 support_count、gain_score，并轻微提高 confidence。

rejected

代表该记忆误导、过期或不适用。会增加 contradict_count、降低 gain_score，并惩罚 confidence。

ignored

被注入但没有明确使用，不应自动奖励。这样避免“进入 prompt 就持续加权”的反馈污染。

显式反馈和隐式反馈的区别

显式反馈来自 adopted_ids / rejected_ids，并要求这些 id 必须属于本轮注入 cohort，且不能同时被采用和拒绝。隐式正反馈只有在很窄的安全条件下才触发：任务成功、置信度高、只注入了一条 memory、且属于 allow-list signal。多个候选混杂时，系统倾向于只记录报告，不做奖励。

08 / Loop Engineering

Loop 到底用在哪里：它横跨任务过程，但不替代记忆检索。

如果把 Loop 说成“又一个召回算法”，就是错的。当前 AMH 的 Loop 是复杂任务的状态账本：它记录目标、检查点、验证证据、产物和结果，让任务可以被复盘、治理、交接和演进。

LoopRun Ledger

必须反复强调的边界

Loop 不参与 RRF、decay、MMR 的打分公式；Loop 也不会自动变成 MemoryItem。Loop 记录过程证据，MemoryItem 记录长期结论。Loop 的价值在于让复杂任务有可验证闭环，而不是让检索“更玄学”。

1
任务启动时：创建 LoopRun。

memory loop create 记录 goal、trigger、project、cwd、adapter、session_id、budget、context、verification_plan。

2
任务推进时：checkpoint。

memory loop checkpoint 记录阶段性进度、发现、风险、待验证事项。它适合复杂 README 改造、跨文件修复、评测闭环等工作。

3
任务完成时：必须有验证证据。

memory loop complete 当前要求 verification evidence。没有验证结果就不能随便标完成。

4
任务失败时：失败事件进入演进控制。

EvolutionControl 会读取 failed loop events，产生 inspect_failed_loops 这类 review_required 风险信号。

5
任务产物有复用价值时：另写 MemoryItem。

例如本页 HTML 完成后，应该写一条 artifact memory 指向产物路径和用途。Loop 只记录过程，长期知识仍由 MemoryItem 承担。

LoopRun 字段和状态机
字段/状态含义价值
loop_idLoopRun 唯一 ID。让一次复杂任务可定位、可追踪。
goal / trigger任务目标和触发原因。解释为什么开始这次闭环。
project / cwd / adapter / session_id运行环境。避免跨 repo、跨 adapter 复盘时混淆。
budget / context预算和上下文。帮助解释任务为什么收窄或扩展。
verification_plan / verification_results计划怎么验、实际怎么验。完成状态不靠口头承诺，而靠证据。
checkpoints阶段性记录。长任务可恢复、可交接。
artifacts产物列表。连接任务过程和最终文件。
outcome结果。成功、失败、阻塞都能进入后续治理。
memory_candidates可能值得沉淀成 MemoryItem 的候选。提醒 Agent 后续写长期知识，但不自动替它写。
状态机created -> running/failed/cancelled；running -> blocked/failed/completed/cancelled；blocked -> running/failed/cancelled；failed -> running。completed/cancelled 是终态，防止任务状态乱跳。

Loop 的三个真实价值

对执行：复杂任务有目标、检查点和验证计划，不会只靠聊天上下文撑着。对治理：失败、阻塞、验证缺失可以被审计和演进控制读取。对协作：不同 Agent 承接同一任务时，不必从零猜“之前做到哪一步”。

09 / Capability Map

从指标、治理、协作三个维度看 AMH。

这部分把能力地图重新压成三条清晰主线，避免“概念很多但重点不突出”。指标回答“怎么量化”，治理回答“怎么可信”，协作回答“怎么跨智能体复用”。

三维能力图

指标 Metrics

  • 召回命中：BM25 rank、vector rank、RRF score、trace。
  • 质量反馈：support、contradict、gain、confidence。
  • 上下文成本：locator/overview/detail token 估算。
  • 链路评测：retrieval ablation、stale hit rate、token cost。
  • 成熟度：source completeness、validation evidence、access、refs。

治理 Governance

  • 来源边界：refs、source、validity、evidence 层分离。
  • 注入门禁：context firewall、sensitivity、review、stale、supersession。
  • 生命周期：decay、retention、maturity、summary rewrite。
  • 冲突控制：contradict、negative feedback、duplicate cluster。
  • 演进控制：failed loops、review_required signals。

协作 Collaboration

  • 多智能体共享：统一 items、sources、index 和 MCP/CLI 接口。
  • 多适配器：Codex、Claude Code、Wukong、Qoder 等适配边界。
  • 承接机制：brief、detail_uri、handoff、Loop checkpoints。
  • 产物复用：artifact memory 指向 README、架构图、验证报告。
  • 人机协同：显式 adopted/rejected 反馈，避免自动强化污染。
10 / Glossary

术语字典：每个名词只说它真正负责的事。

这一段适合读者卡住时查。重点是边界：一个名词负责什么，不负责什么。

查漏补缺
Agent Memory Hub

多智能体协作的共享记忆系统。核心不是存聊天，而是把证据、长期条目、索引、召回、注入和治理拆层。

Brain Pool

共享大脑池。未来 Agent 通过 search/brief/read 等接口拉取 MemoryItem，而不是指定“发给某个 Agent”。

MemoryItem

长期可复用结论。写入 items/，有 type、title、summary、refs、confidence、context_views 等字段。

Source / Evidence

原始证据层。保存 prompt、transcript、文件、资源、运行事件等来源，不默认等同于长期记忆。

ContextViews

一个 item 的多层阅读视图：locator 最短，overview 中等，detail_uri 指向正文。

FTS

Full-Text Search，全文检索。AMH 用 SQLite FTS5 做关键词/全文匹配。

BM25

全文检索排序函数。适合精确词、路径、命令、错误文本和专有名词。

Vector Search

向量语义检索。适合表达不同但语义接近的问题。

RRF

Reciprocal Rank Fusion，倒数排名融合。把 BM25 和 vector 的排名加到同一候选分。

Rerank

可选精排。用 cross-encoder 更细判断 query 和候选文本是否匹配。

Confidence

item 的可信程度。默认 0.7，受访问、确认、拒绝、替代等反馈影响。

Decay

遗忘衰减。不同记忆类型有不同半衰期，避免过期 signal/handoff 长期占据高位。

Feedback Value

历史使用价值乘子。support 增益，contradict 惩罚，gain_score 放大或削弱。

Maturity

成熟度。衡量来源、置信、反馈、访问、验证证据等；用于治理和装载，不是主召回乘法项。

MMR

Maximal Marginal Relevance，多样性重排。避免一批重复 item 同时进入上下文。

Hopfield Expansion

可选关联扩展。由第一跳候选形成 attractor，再寻找语义邻居。默认不等于主检索。

Graph Expansion

基于 refs_graph 的引用邻居扩展。补齐与命中 item 有来源关系的条目。

Context Firewall

注入前防火墙。处理敏感、过期、低置信、缺来源、负反馈、主题覆盖不足等风险。

ContextPack

最终要注入给模型的上下文包，包含选中的 view、text、detail_uri、token 估算和读取提示。

LoopRun

复杂任务账本。记录目标、检查点、验证、产物和结果。不是排序算法，不自动写 MemoryItem。

Handoff

交接型记忆。用于新会话承接，但属于更易过期的 ephemeral 类，需要防止旧 handoff 误导。

Artifact Memory

产物型记忆。记录 README、报告、架构图、验证结果等文件路径和用途，不复制整个产物内容。
